# Feature Implementation Plan: Multi-Account Strategy Layer

**Feature**: Add an "account" abstraction layer so multiple strategies can run on separate IG accounts concurrently. Each account has a strategy with a free-form markdown `prompt` that drives tick execution.
**GitHub Issue**: N/A
**Created**: 2026-04-01
**Status**: Complete
**Complexity**: Medium (1-2 hours)

---

## Overview

Currently the trading bot runs a single strategy on a single IG account per tick. The config (`BotConfig`) has one `strategy` field, one set of credentials, and one watchlist. To support running multiple different strategies on separate IG accounts concurrently, we need to introduce two new database-persisted entities:

1. **`strategies`** table ‚Äî Stores named strategy configurations with a markdown `prompt` column. The prompt is free-form but has a standardized metadata header (tickers, risk params, strategy type, etc.).
2. **`accounts`** table ‚Äî Stores per-account config (IG credentials reference, account name, active strategy FK). Each account maps to one IG account and one strategy.

The tick orchestrator will then iterate over active accounts, load each account's strategy prompt, and execute accordingly.

## Design Decisions

### Strategy Prompt Format

The `prompt` column stores markdown with a YAML frontmatter header:

```markdown
---
name: "FTSE Trend Follower"
tickers:
  - epic: "IX.D.FTSE.DAILY.IP"
    expiry: "DFB"
    currencyCode: "GBP"
  - epic: "IX.D.DAX.DAILY.IP"
    expiry: "DFB"
    currencyCode: "EUR"
strategyType: "trend-following"
riskPerTrade: 0.01
maxOpenPositions: 3
---

## Trading Rules

Buy when SMA10 crosses above SMA20 and price pulls back within 1 ATR.
Sell when the opposite crossover occurs.

Use wider stops (2x ATR) during high-volatility periods.

## Special Conditions

- Avoid trading during major economic announcements
- Reduce position size by 50% on Fridays
```

This gives structure where needed (tickers, strategy type) while allowing completely free-form instructions in the markdown body. The frontmatter is parseable, the body is passed to the LLM or strategy engine.

### Account Model

Each account represents one "bot instance" ‚Äî it has:
- A reference name (e.g. "UK Indices Demo")
- IG API credentials (apiKey, username, encrypted password reference)
- A link to a strategy (FK to strategies table)
- Its own circuit breaker state, risk config, and interval
- An `isActive` flag to enable/disable without deleting

### Scoping Existing Tables

All existing tables (ticks, signals, trades, positions) will gain an `accountId` column to scope data per-account. This is added as **nullable** initially to avoid breaking the existing migration or data. New rows will always have it set; old rows will have NULL (meaning "legacy/default account").

## Final Notes

The key architectural decision is keeping the `accountId` column **nullable** on existing tables. This ensures:
1. Existing data (legacy single-account mode) continues to work
2. No breaking changes to the existing migration
3. Gradual adoption ‚Äî the tick orchestrator can be refactored separately
4. Tests don't break because all existing insert operations omit `accountId` (defaults to NULL)

The strategy `prompt` format uses YAML frontmatter because:
1. It's a well-established pattern (Hugo, Jekyll, MDX, etc.)
2. The structured header (tickers, risk config) can be parsed and validated
3. The markdown body remains completely free-form for LLM consumption
4. No heavy dependencies needed ‚Äî simple regex + JSON-like parsing
